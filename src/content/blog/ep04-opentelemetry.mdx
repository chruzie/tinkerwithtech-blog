# OpenTelemetry in 7 Minutes

Here's the silent tax every observability stack accumulates.

Prometheus client library in the app. Jaeger agent sidecar. Loki log shipper. Three SDKs to maintain, three backends to configure, three dashboards to check when something breaks — and no way to correlate a trace with a metric spike unless you've wired up exemplars by hand.

OpenTelemetry eliminates this. One SDK. One wire protocol (OTLP). One Collector that receives all three signals and routes them to whatever backends you already use. Your Prometheus, Grafana, and Jaeger setups stay unchanged — you just stop maintaining three separate instrumentation libraries.

This post walks through a complete homelab deployment: OTel Collector with a multi-signal pipeline, Prometheus with remote-write receiver, Grafana, and a sample app emitting real OTLP data — all on a local k3d cluster, no cloud account needed.

## Architecture: The OTel Collector Pipeline

```
Your App (any language)
      │
      │  OTLP (gRPC :4317 or HTTP :4318)
      ▼
┌─────────────────────────────────────────┐
│          OTel Collector                 │
│                                         │
│  Receivers:   otlp, prometheus          │
│  Processors:  memory_limiter, batch     │
│  Exporters:   debug, prometheusremote   │
│                                         │
│  Pipelines:                             │
│    traces:  [otlp] → [batch] → [debug]  │
│    metrics: [otlp,prom] → [batch]       │
│                → [debug, promremote]    │
│    logs:    [otlp] → [batch] → [debug]  │
└────────────────┬────────────────────────┘
                 │
        remote_write (HTTP)
                 │
                 ▼
          Prometheus ──→ Grafana
```

The Collector is the **central bus**. It decouples your application instrumentation from your observability backends. Swap backends — Jaeger instead of debug for traces, Loki for logs — without changing a single line of application code.

### Why the Collector matters

Without OTel: your app speaks Prometheus format directly, or runs a Jaeger agent sidecar, or a Fluentd log shipper. Each integration has its own config format, its own failure mode, and its own upgrade path.

With OTel: your app speaks OTLP to the Collector. The Collector translates. Your backends evolve independently. The Collector itself is stateless and horizontally scalable.

## Step-by-Step: Full Homelab Setup

### Step 1 — Create the k3d cluster

```bash
k3d cluster create otel --agents 1 --wait \
  --k3s-arg "--disable=traefik@server:0"
kubectl config rename-context k3d-otel k3d-otel 2>/dev/null || true
kubectl cluster-info --context k3d-otel
```

### Step 2 — Create namespaces

```bash
kubectl create namespace observability --context k3d-otel
kubectl create namespace sample-app --context k3d-otel
```

### Step 3 — Install kube-prometheus-stack (Prometheus + Grafana)

```bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

helm install kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  --namespace observability \
  --kube-context k3d-otel \
  --set grafana.adminPassword="tinkerwithtech" \
  --set prometheus.prometheusSpec.scrapeInterval="15s" \
  --set prometheus.prometheusSpec.evaluationInterval="15s" \
  --set prometheus.prometheusSpec.enableRemoteWriteReceiver=true \
  --set alertmanager.enabled=false \
  --set nodeExporter.enabled=false \
  --set kubeStateMetrics.enabled=false \
  --wait --timeout=300s
```

**Important:** `enableRemoteWriteReceiver=true` is required. Without it, the OTel Collector's `prometheusremotewrite` exporter will fail with a 404 — Prometheus rejects remote write requests by default.

### Step 4 — Install the OTel Collector

```bash
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
helm repo update

helm install otel-collector open-telemetry/opentelemetry-collector \
  --namespace observability \
  --kube-context k3d-otel \
  --set mode=deployment \
  --set fullnameOverride=otel-collector \
  --set image.repository=otel/opentelemetry-collector-contrib \
  --values - <<'HELM_VALUES'
config:
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317
        http:
          endpoint: 0.0.0.0:4318
    prometheus:
      config:
        scrape_configs:
          - job_name: "otel-collector-self"
            scrape_interval: 10s
            static_configs:
              - targets: ["localhost:8888"]

  processors:
    batch:
      timeout: 5s
      send_batch_size: 512
    memory_limiter:
      check_interval: 1s
      limit_mib: 250

  exporters:
    debug:
      verbosity: detailed
    prometheusremotewrite:
      endpoint: "http://kube-prometheus-stack-prometheus.observability.svc.cluster.local:9090/api/v1/write"
      tls:
        insecure: true

  service:
    pipelines:
      traces:
        receivers: [otlp]
        processors: [memory_limiter, batch]
        exporters: [debug]
      metrics:
        receivers: [otlp, prometheus]
        processors: [memory_limiter, batch]
        exporters: [debug, prometheusremotewrite]
      logs:
        receivers: [otlp]
        processors: [memory_limiter, batch]
        exporters: [debug]

ports:
  otlp:
    enabled: true
    containerPort: 4317
    servicePort: 4317
    protocol: TCP
  otlp-http:
    enabled: true
    containerPort: 4318
    servicePort: 4318
    protocol: TCP
  metrics:
    enabled: true
    containerPort: 8888
    servicePort: 8888
    protocol: TCP
HELM_VALUES
```

**Note:** `image.repository=otel/opentelemetry-collector-contrib` is required since chart v0.110+. The contrib distribution includes the `prometheusremotewrite` exporter. Core distribution does not.

Wait for rollout:

```bash
kubectl rollout status deployment/otel-collector -n observability --context k3d-otel
```

### Step 5 — Deploy the sample app

A minimal load generator that sends OTLP traces and metrics every 10 seconds — no language SDK required, just raw HTTP to the OTLP endpoint:

```bash
kubectl apply --context k3d-otel -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-app
  namespace: sample-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: sample-app
  template:
    metadata:
      labels:
        app: sample-app
    spec:
      containers:
        - name: load-generator
          image: curlimages/curl:latest
          command:
            - /bin/sh
            - -c
            - |
              COUNT=0
              while true; do
                COUNT=$((COUNT + 1))
                NOW_NS=$(date +%s)000000000
                START_NS=$(( $(date +%s) - 1 ))000000000
                curl -s -X POST http://otel-collector.observability.svc.cluster.local:4318/v1/traces \
                  -H "Content-Type: application/json" \
                  -d "{\"resourceSpans\":[{\"resource\":{\"attributes\":[{\"key\":\"service.name\",\"value\":{\"stringValue\":\"sample-app\"}}]},\"scopeSpans\":[{\"scope\":{\"name\":\"sample-app-tracer\"},\"spans\":[{\"traceId\":\"0102030405060708090a0b0c0d0e0f10\",\"spanId\":\"010203040506070${COUNT}\",\"name\":\"GET /api/users\",\"kind\":2,\"startTimeUnixNano\":\"${START_NS}\",\"endTimeUnixNano\":\"${NOW_NS}\",\"status\":{\"code\":1}}]}]}]}" || true
                curl -s -X POST http://otel-collector.observability.svc.cluster.local:4318/v1/metrics \
                  -H "Content-Type: application/json" \
                  -d "{\"resourceMetrics\":[{\"resource\":{\"attributes\":[{\"key\":\"service.name\",\"value\":{\"stringValue\":\"sample-app\"}}]},\"scopeMetrics\":[{\"scope\":{\"name\":\"sample-app-metrics\"},\"metrics\":[{\"name\":\"http_requests_total\",\"sum\":{\"dataPoints\":[{\"attributes\":[{\"key\":\"http.method\",\"value\":{\"stringValue\":\"GET\"}}],\"startTimeUnixNano\":\"${START_NS}\",\"timeUnixNano\":\"${NOW_NS}\",\"asInt\":${COUNT}}],\"aggregationTemporality\":2,\"isMonotonic\":true}}]}]}]}" || true
                curl -s -X POST http://otel-collector.observability.svc.cluster.local:4318/v1/logs \
                  -H "Content-Type: application/json" \
                  -d "{\"resourceLogs\":[{\"resource\":{\"attributes\":[{\"key\":\"service.name\",\"value\":{\"stringValue\":\"sample-app\"}}]},\"scopeLogs\":[{\"scope\":{\"name\":\"sample-app-logger\"},\"logRecords\":[{\"timeUnixNano\":\"${NOW_NS}\",\"severityText\":\"INFO\",\"body\":{\"stringValue\":\"handled GET /api/users\"},\"attributes\":[{\"key\":\"http.status_code\",\"value\":{\"intValue\":200}}]}]}]}]}" || true
                sleep 10
              done
          env:
            - name: OTEL_EXPORTER_OTLP_ENDPOINT
              value: "http://otel-collector.observability.svc.cluster.local:4318"
            - name: OTEL_SERVICE_NAME
              value: "sample-app"
EOF

kubectl rollout status deployment/sample-app -n sample-app --context k3d-otel
```

### Step 6 — Start port-forwards

```bash
kubectl port-forward -n observability svc/kube-prometheus-stack-grafana 3000:80 \
  --context k3d-otel &
kubectl port-forward -n observability svc/kube-prometheus-stack-prometheus 9090:9090 \
  --context k3d-otel &
kubectl port-forward -n observability svc/otel-collector 4318:4318 \
  --context k3d-otel &
```

> The `&` runs each port-forward in the background of your current shell session. They terminate when you close the terminal. Re-run the commands above in a new session if needed.

### The HERO MOMENT — Three Signals, One Process

```bash
kubectl logs -n observability \
  -l app.kubernetes.io/name=opentelemetry-collector \
  --context k3d-otel --tail=50
```

You'll see all three signals flowing through the debug exporter:

```
info  Traces  {"otelcol.signal": "traces", "resource spans": 1, "spans": 1}
     -> service.name: Str(sample-app)

Span #0  Name: GET /api/users

info  Metrics  {"otelcol.signal": "metrics", "resource metrics": 2, "metrics": 1}
     -> service.name: Str(sample-app)
```

Traces received from the OTLP receiver. Metrics batched and forwarded to Prometheus via remote write. Logs flowing through the pipeline. **Three signals. One process. One config file.**

## What You Built

```
k3d cluster: otel
└── observability/
    ├── otel-collector          ← Central signal bus (OTLP in, debug + promremote out)
    ├── prometheus              ← Stores metrics (remote-write receiver enabled)
    └── grafana                 ← Visualises Prometheus metrics

sample-app/
└── sample-app                  ← curl container; sends OTLP traces + metrics every 10s

Dashboards:
  Grafana:     http://localhost:3000  (admin / tinkerwithtech)
  Prometheus:  http://localhost:9090
```

**Key architectural facts to remember:**

| Concept | Detail |
|---------|--------|
| OTLP | The standard wire protocol for all three signals — replace any backend without app changes |
| OTel Collector | Stateless, scalable — runs as DaemonSet for logs, Deployment for traces/metrics |
| `image.repository` | Must be `otel/opentelemetry-collector-contrib` for `prometheusremotewrite` exporter |
| Remote-write receiver | Must enable on Prometheus: `--web.enable-remote-write-receiver` (or Helm flag) |
| OTel activity | #2 most active CNCF project — SDK support spans 11+ languages |
