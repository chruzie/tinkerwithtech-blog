GitOps has an awkward gap: secrets.

You can put every `Deployment`, `Service`, and `ConfigMap` in Git and sync it with Argo CD. But the moment you need a database password or an API key, you're either committing a sealed/encrypted blob, reaching through a CSI driver at pod start, or typing `kubectl create secret` by hand outside the GitOps flow.

The External Secrets Operator (ESO) closes that gap. Your Git repo contains an `ExternalSecret` that says *"populate a `Secret` named `myapp-secret` with keys pulled from `secret/demo/myapp` in Vault"*. ESO watches Vault, materializes the `Secret` into the cluster, and re-syncs when the source rotates. No plaintext in Git. No out-of-band `kubectl`. No sealed-secret rewrapping when the key expires.

This post sets up Vault in dev mode, installs ESO, wires them together, and demonstrates a live rotation — all on k3d.

---

## What's being solved

There are three established patterns for getting secrets into a pod. All have tradeoffs:

| Pattern | Example | Problem |
|---------|---------|---------|
| Commit plaintext | `kubectl apply secret.yaml` | Obvious |
| Commit encrypted | SealedSecrets, SOPS | Rewrap on every rotation; encrypted value in Git is still in Git |
| Mount at runtime | Vault Agent / CSI Secrets Store | Pod tightly coupled to the driver; breaks for anything that reads `/var/run/secrets` or env vars |

ESO is a fourth pattern: **the Secret still lives in the cluster, but its source of truth is external**. Your app consumes a normal `Secret` (env var or volume). Your Git repo describes *where the value comes from*, not what it is. Rotations happen at the source; ESO reconciles.

The same pattern works for AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, 1Password, Doppler, Pulumi ESC, and 20+ other backends. Only the `SecretStore` spec changes.

## Architecture

```
k3d cluster: ep10-secrets
│
├── vault namespace
│   └── vault-0 (dev mode, root token = "root")
│       └── KV v2 engine at path "secret/"
│            └── secret/demo/myapp  { username, password, api_key }
│
├── external-secrets namespace
│   ├── external-secrets              (operator)
│   ├── external-secrets-webhook      (validates CRDs)
│   └── external-secrets-cert-controller
│
└── demo namespace
    ├── vault-token       (Secret — holds Vault token for auth)
    ├── vault-backend     (SecretStore — points ESO at Vault)
    ├── myapp-secret      (ExternalSecret — declares what to sync)
    ├── myapp-secret      (Secret — materialized by ESO)
    └── consumer          (Pod — consumes Secret as env vars)
```

The reconcile loop:

1. ESO reads the `ExternalSecret` resource.
2. Resolves the `SecretStore` reference to find the backend (Vault).
3. Authenticates to Vault using `tokenSecretRef` (or Kubernetes auth, or AppRole, or AWS IAM, etc.).
4. Fetches `secret/demo/myapp`, extracts the requested properties.
5. Materializes a `Secret` with those values. Sets `ExternalSecret.status.syncedResourceVersion`.
6. Every `refreshInterval`, repeats — so rotations at the source propagate into the cluster automatically.

## Step-by-step

<Step number="1" title="Create the k3d cluster" time="~30 sec">

```bash
k3d cluster create ep10-secrets \
  --agents 1 \
  --wait \
  --k3s-arg "--disable=traefik@server:0"
```

<ExpectedOutput>
```
INFO[0000] Created image volume k3d-ep10-secrets-images
INFO[0001] Creating node 'k3d-ep10-secrets-server-0'
INFO[0001] Creating node 'k3d-ep10-secrets-agent-0'
INFO[0016] Cluster 'ep10-secrets' created successfully!
```
</ExpectedOutput>

</Step>

<Step number="2" title="Install Vault in dev mode" time="~45 sec">

Dev mode auto-initializes Vault, auto-unseals it, enables a KV v2 engine at `secret/`, and fixes the root token so the demo is deterministic.

```bash
helm repo add hashicorp https://helm.releases.hashicorp.com
helm repo update hashicorp

helm install vault hashicorp/vault \
  --namespace vault --create-namespace \
  --version 0.32.0 \
  --set "server.dev.enabled=true" \
  --set "server.dev.devRootToken=root" \
  --set "injector.enabled=false" \
  --kube-context k3d-ep10-secrets \
  --wait --timeout=300s
```

Wait for the Vault pod to become Ready:

```bash
kubectl wait --for=condition=Ready pod/vault-0 \
  -n vault --context k3d-ep10-secrets --timeout=180s
```

<ExpectedOutput>
```
pod/vault-0 condition met
NAME      READY   STATUS    RESTARTS   AGE
vault-0   1/1     Running   0          16s
```
</ExpectedOutput>

Confirm the KV engine is mounted:

```bash
kubectl exec -n vault vault-0 --context k3d-ep10-secrets -- \
  sh -c 'vault login root >/dev/null && vault secrets list'
```

<ExpectedOutput>
```
Path          Type         Description
----          ----         -----------
cubbyhole/    cubbyhole    per-token private secret storage
identity/     identity     identity store
secret/       kv           key/value secret storage
sys/          system       system endpoints used for control, policy and debugging
```
</ExpectedOutput>

<Callout type="info">
The `injector.enabled=false` setting disables the Vault Agent Injector (a sidecar-based approach to mounting secrets). We don't need it — ESO pulls from Vault directly.
</Callout>

</Step>

<Step number="3" title="Write a secret into Vault" time="~5 sec">

```bash
kubectl exec -n vault vault-0 --context k3d-ep10-secrets -- \
  sh -c 'vault login root >/dev/null && \
    vault kv put secret/demo/myapp \
      username=admin \
      password=s3cr3t-from-vault \
      api_key=AKIAPLACEHOLDER'
```

<ExpectedOutput>
```
===== Secret Path =====
secret/data/demo/myapp

======= Metadata =======
Key                Value
---                -----
created_time       2026-04-23T00:14:05Z
version            1
```
</ExpectedOutput>

The KV v2 engine stores secrets at `secret/data/<path>` internally (that's the `===== Secret Path =====` line). ESO references the logical path `demo/myapp`.

</Step>

<Step number="4" title="Install External Secrets Operator" time="~45 sec">

```bash
helm repo add external-secrets https://charts.external-secrets.io
helm repo update external-secrets

helm install external-secrets external-secrets/external-secrets \
  --namespace external-secrets --create-namespace \
  --version 2.3.0 \
  --set installCRDs=true \
  --kube-context k3d-ep10-secrets \
  --wait --timeout=300s
```

```bash
kubectl get pods -n external-secrets --context k3d-ep10-secrets
```

<ExpectedOutput>
```
NAME                                                READY   STATUS    RESTARTS   AGE
external-secrets-586cd4877b-wxhnx                   1/1     Running   0          33s
external-secrets-cert-controller-6d59685bfb-8rzlz   1/1     Running   0          33s
external-secrets-webhook-7d895cc85d-dslsg           1/1     Running   0          33s
```
</ExpectedOutput>

Three pods: the operator (reconciles `ExternalSecret` resources), the cert controller (manages webhook TLS), and the webhook (validates specs).

</Step>

<Step number="5" title="Create a SecretStore pointing to Vault" time="~10 sec">

```bash
kubectl create namespace demo --context k3d-ep10-secrets

# The Vault root token has to live somewhere ESO can read.
# In production this is a Kubernetes ServiceAccount token or IAM role — not a static token.
kubectl create secret generic vault-token \
  -n demo --context k3d-ep10-secrets \
  --from-literal=token=root

kubectl apply --context k3d-ep10-secrets -f - <<'EOF'
apiVersion: external-secrets.io/v1
kind: SecretStore
metadata:
  name: vault-backend
  namespace: demo
spec:
  provider:
    vault:
      server: "http://vault.vault.svc.cluster.local:8200"
      path: "secret"
      version: "v2"
      auth:
        tokenSecretRef:
          name: vault-token
          key: token
EOF
```

Verify the store is valid and connected:

```bash
kubectl get secretstore vault-backend -n demo --context k3d-ep10-secrets
```

<ExpectedOutput>
```
NAME            AGE   STATUS   CAPABILITIES   READY
vault-backend   7s    Valid    ReadWrite      True
```
</ExpectedOutput>

`READY=True` means ESO successfully authenticated to Vault with the token. If this says `False`, check that the Vault URL resolves in-cluster and the token in `vault-token` is correct.

<Callout type="tip">
`SecretStore` is namespace-scoped. For a store that multiple namespaces can reference, use `ClusterSecretStore` — same spec, no `metadata.namespace`.
</Callout>

</Step>

<Step number="6" title="Declare the ExternalSecret" time="~5 sec">

This is the Git-committable resource. It maps Vault keys → k8s Secret keys.

```bash
kubectl apply --context k3d-ep10-secrets -f - <<'EOF'
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: myapp-secret
  namespace: demo
spec:
  refreshInterval: 30s
  secretStoreRef:
    name: vault-backend
    kind: SecretStore
  target:
    name: myapp-secret
    creationPolicy: Owner
  data:
    - secretKey: username
      remoteRef:
        key: demo/myapp
        property: username
    - secretKey: password
      remoteRef:
        key: demo/myapp
        property: password
    - secretKey: api_key
      remoteRef:
        key: demo/myapp
        property: api_key
EOF
```

Check sync status:

```bash
kubectl get externalsecret myapp-secret -n demo --context k3d-ep10-secrets
kubectl get secret          myapp-secret -n demo --context k3d-ep10-secrets
```

<ExpectedOutput>
```
NAME           STORETYPE     STORE           REFRESH INTERVAL   STATUS         READY   LAST SYNC
myapp-secret   SecretStore   vault-backend   30s                SecretSynced   True    3s

NAME           TYPE     DATA   AGE
myapp-secret   Opaque   3      3s
```
</ExpectedOutput>

Decode the materialized `Secret` to confirm ESO pulled the right values:

```bash
for k in username password api_key; do
  printf "%-10s = " "$k"
  kubectl get secret myapp-secret -n demo --context k3d-ep10-secrets \
    -o jsonpath="{.data.$k}" | base64 -d
  echo
done
```

<ExpectedOutput>
```
username   = admin
password   = s3cr3t-from-vault
api_key    = AKIAPLACEHOLDER
```
</ExpectedOutput>

</Step>

<Step number="7" title="Consume the Secret from a Pod" time="~15 sec">

Nothing about the consumer pod knows about ESO or Vault. It just reads a normal Secret.

```bash
kubectl apply --context k3d-ep10-secrets -n demo -f - <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: consumer
spec:
  containers:
    - name: app
      image: busybox:1.37
      command: ["/bin/sh", "-c", "sleep 3600"]
      envFrom:
        - secretRef:
            name: myapp-secret
EOF

kubectl wait --for=condition=Ready pod/consumer \
  -n demo --context k3d-ep10-secrets --timeout=60s

kubectl exec -n demo consumer --context k3d-ep10-secrets -- \
  env | grep -E '^(username|password|api_key)='
```

<ExpectedOutput>
```
api_key=AKIAPLACEHOLDER
password=s3cr3t-from-vault
username=admin
```
</ExpectedOutput>

</Step>

<Step number="8" title="Rotate the secret and watch it propagate" time="~40 sec">

This is the payoff. Rotate the value at the source:

```bash
kubectl exec -n vault vault-0 --context k3d-ep10-secrets -- \
  sh -c 'vault login root >/dev/null && \
    vault kv put secret/demo/myapp \
      username=admin \
      password=NEW-ROTATED-VALUE \
      api_key=AKIAPLACEHOLDER'
```

Vault bumps the version to `2`. Wait for the refresh interval (30s):

```bash
sleep 35

kubectl get secret myapp-secret -n demo --context k3d-ep10-secrets \
  -o jsonpath='{.data.password}' | base64 -d
echo
```

<ExpectedOutput>
```
NEW-ROTATED-VALUE
```
</ExpectedOutput>

<Callout type="check">
The Kubernetes Secret picked up the new value within ~30 seconds. No `kubectl apply`, no `helm upgrade`, no Argo CD sync. The pod's env vars are a point-in-time snapshot, so the consumer still needs a rolling restart to pick up the change — but the Secret itself is current. Projected volumes are re-read on kubelet sync (usually ~60s), so mounting the Secret as a volume gives you hot rotation with no restart.
</Callout>

</Step>

## Going to production

Three things change when you move off dev mode:

### 1. Vault auth

The `tokenSecretRef` we used is a static bearer token. Real deployments use one of:

- **Kubernetes auth** — ESO pod's ServiceAccount JWT is verified by Vault against the cluster. Tokens are short-lived and auto-rotated.
- **AppRole** — ESO holds a `role_id` + `secret_id` pair. The `secret_id` can be wrapped for single-use delivery.
- **AWS/GCP/Azure IAM** — ESO authenticates using the cloud identity of the node/pod.

For Kubernetes auth, the `SecretStore` spec becomes:

```yaml
auth:
  kubernetes:
    mountPath: kubernetes
    role: my-role
    serviceAccountRef:
      name: external-secrets
```

### 2. Vault TLS and HA

Dev mode is HTTP and in-memory. Production Vault runs as a 3+ node StatefulSet with Raft storage, TLS on the listener, and auto-unseal via a cloud KMS.

### 3. Least-privilege policies

The root token can read every secret. Real deployments bind each `SecretStore` to a Vault role with a policy limited to the paths that store's consumers need — typically `secret/data/<team>/*` or `secret/data/<env>/<app>/*`.

## What's next

Two natural follow-ons:

- **Swap Vault for your cloud's secret manager.** Only the `SecretStore` spec changes — the `ExternalSecret` and the consuming Pod stay identical. That's the portability payoff.
- **Layer ESO under Argo CD.** Your `ExternalSecret` resources go in Git with your deployments. Argo syncs them. ESO materializes the `Secret`. Now the full app, including credentials, is declarative from a single repo — with the actual secret material never leaving Vault.
