Here's the question every platform team eventually faces.

You've been using Terraform for years. It works. But now developers want self-service infrastructure — apply a YAML file and get a database, without triggering a pipeline or checking a Slack notification. They want the Kubernetes experience for everything.

That's where Crossplane comes in. But "replace Terraform with Crossplane" is the wrong framing. They are fundamentally different tools solving different problems. This post runs both on a local k3d cluster — no cloud account needed — and shows you exactly when to use which.

---

## Prerequisites

| Tool | Version | Install |
|------|---------|---------|
| `kubectl` | any | `brew install kubectl` |
| `k3d` | 5+ | `brew install k3d` |
| `helm` | 3+ | `brew install helm` |
| `terraform` | 1.5+ | `brew install terraform` |
| Docker | any | [docker.com](https://www.docker.com) |

The demo uses the **Crossplane NOP provider** (no-op — behaves exactly like a real managed resource without touching cloud APIs) and **Terraform's null provider**. Both run air-gapped with zero cloud credentials.

## Architecture

### The Terraform model — pipeline

```diagram
Developer ──(HCL file)──► terraform apply ──(API call)──► Cloud Resource Created
                          (manual trigger)                         │
                                                          state stored
                                                                   │
                                                        terraform.tfstate
                                                                   │
                                               drift invisible until next plan
                                                                   ▼
                                                      ⚠ Config drift — nobody notices
                                                      (hours or days before detection)
```

Terraform is a **pipeline tool**. You invoke it, it provisions, it stores state, then it waits until you run it again. Drift is invisible until the next `terraform plan`. In a busy team, that gap can be hours or days.

### The Crossplane model — control plane

```diagram
Developer
   │  kubectl apply YAML CR
   ▼
Kubernetes API Server
   │  watch event
   ▼
Crossplane Controller ◄──────────────────────┐
   │  reconcile                               │
   ▼                                          │
Cloud API / NOP provider                      │ continuous watch
   │                                          │
   ▼                                          │
Resource: Ready + Synced ─────────────────────┘
   │
   │  drift detected?
   ▼
Auto-correct in milliseconds
```

Crossplane runs a **Kubernetes controller** inside your cluster. It watches Custom Resource objects and continuously reconciles the real world to match them. Drift is not a scheduled event — it's detected and corrected in the same reconciliation loop that fires on every watch event.

## Step-by-step

<Step number="1" title="Create the k3d cluster" time="~1 min">

```bash
k3d cluster create crossplane --agents 1 --wait \
  --k3s-arg "--disable=traefik@server:0"
```

```bash
kubectl cluster-info --context k3d-crossplane
```

<Callout type="check">
`kubectl get nodes --context k3d-crossplane` shows 2 nodes.
</Callout>

</Step>

<Step number="2" title="Install Crossplane via Helm" time="~3 min">

```bash
helm repo add crossplane-stable https://charts.crossplane.io/stable
helm repo update

helm install crossplane crossplane-stable/crossplane \
  --namespace crossplane-system \
  --create-namespace \
  --wait \
  --kube-context k3d-crossplane
```

```bash
kubectl rollout status deployment/crossplane -n crossplane-system --context k3d-crossplane
kubectl rollout status deployment/crossplane-rbac-manager -n crossplane-system --context k3d-crossplane
```

<ExpectedOutput>
```
deployment "crossplane" successfully rolled out
deployment "crossplane-rbac-manager" successfully rolled out
```
</ExpectedOutput>

</Step>

<Step number="3" title="Install the NOP provider" time="~2 min">

The NOP provider behaves exactly like a real Crossplane provider — it accepts managed resources, marks them `Ready+Synced` after a configurable delay, and responds to deletion — without making any cloud API calls.

```bash
kubectl apply --context k3d-crossplane -f - <<'EOF'
apiVersion: pkg.crossplane.io/v1
kind: Provider
metadata:
  name: provider-nop
spec:
  package: xpkg.upbound.io/crossplane-contrib/provider-nop:v0.3.0
  packagePullPolicy: IfNotPresent
EOF
```

```bash
kubectl get provider provider-nop --context k3d-crossplane -w
# Wait for HEALTHY=True, then Ctrl+C
```

<ExpectedOutput>
```
NAME           INSTALLED   HEALTHY   PACKAGE                                                  AGE
provider-nop   True        True      xpkg.upbound.io/crossplane-contrib/provider-nop:v0.3.0   90s
```
</ExpectedOutput>

</Step>

<Step number="4" title="Run the Terraform demo" time="~2 min">

This workspace simulates provisioning a cloud resource using `null_resource` with a `local-exec` provisioner. In production you'd use `aws_s3_bucket`, `google_storage_bucket`, etc.

```bash
mkdir -p /tmp/ep03-terraform-demo
cat > /tmp/ep03-terraform-demo/main.tf <<'TFEOF'
terraform {
  required_providers {
    null  = { source = "hashicorp/null",  version = "~> 3.0" }
    local = { source = "hashicorp/local", version = "~> 2.0" }
  }
}

variable "environment" { default = "production" }
variable "team"        { default = "platform" }

resource "null_resource" "simulated_bucket" {
  triggers = { environment = var.environment, team = var.team }

  provisioner "local-exec" {
    command = <<-CMD
      mkdir -p /tmp/ep03-bucket
      echo '{"environment":"${var.environment}","team":"${var.team}","created_by":"terraform"}' \
        > /tmp/ep03-bucket/config.json
    CMD
  }

  provisioner "local-exec" {
    when    = destroy
    command = "rm -rf /tmp/ep03-bucket"
  }
}

resource "local_file" "state_marker" {
  content  = "terraform-managed: ${var.environment}"
  filename = "/tmp/ep03-bucket/terraform.marker"
}
TFEOF

cd /tmp/ep03-terraform-demo && terraform init -input=false && terraform apply -auto-approve -input=false && cd -
```

<ExpectedOutput>
```
null_resource.simulated_bucket: Creating...
null_resource.simulated_bucket: Provisioning with 'local-exec'...
local_file.state_marker: Creating...
Apply complete! Resources: 2 added, 0 changed, 0 destroyed.
```
</ExpectedOutput>

</Step>

<Step number="5" title="Demonstrate Terraform's drift blindspot" time="~1 min">

Simulate external drift — someone changes the resource outside of Terraform:

```bash
echo '{"environment":"staging","team":"rogue-engineer","created_by":"manual"}' \
  > /tmp/ep03-bucket/config.json
```

Now run `terraform plan` to check if it detects the change:

```bash
cd /tmp/ep03-terraform-demo && terraform plan -input=false && cd -
```

<ExpectedOutput>
```
No changes. Your infrastructure matches the configuration.
```
</ExpectedOutput>

<Callout type="warning">
Terraform's state tracks the *trigger variables* (`environment`, `team`), not the file content itself. The configuration drift is completely invisible. In a real cloud environment this manifests as drift Terraform may never detect, depending on what changed and which attributes the provider refreshes.
</Callout>

</Step>

<Step number="6" title="Apply a Crossplane managed resource" time="~1 min">

```bash
kubectl apply --context k3d-crossplane -f - <<'EOF'
apiVersion: nop.crossplane.io/v1alpha1
kind: NopResource
metadata:
  name: demo-infrastructure
  annotations:
    crossplane.io/external-name: "ep03-simulated-bucket"
spec:
  forProvider:
    conditionAfter:
      - conditionType: Ready
        conditionStatus: "True"
        time: 5s
      - conditionType: Synced
        conditionStatus: "True"
        time: 5s
  writeConnectionSecretToRef:
    namespace: crossplane-system
    name: demo-infrastructure-conn
EOF
```

```bash
kubectl get managed --context k3d-crossplane
```

<ExpectedOutput>
```
NAME                  READY   SYNCED   AGE
demo-infrastructure   True    True     12s
```
</ExpectedOutput>

No pipeline triggered. No backend configured. The controller saw the resource, ran its reconcile function, and the resource is managed.

</Step>

## The Honest Decision Framework

| Need | Tool |
|------|------|
| Continuous drift detection and auto-correction | Crossplane |
| Developer self-service infrastructure via YAML | Crossplane |
| Complex greenfield provisioning (VPCs, accounts) | Terraform |
| Existing team with deep Terraform expertise | Terraform |
| Platform team + GitOps + Kubernetes-native | **Both** |

The most common production pattern: **Terraform provisions the Kubernetes cluster. Crossplane runs inside it and provisions everything else.** They're complementary layers, not competitors.

## Teardown

When you're done experimenting, clean up everything:

```bash
k3d cluster delete crossplane
rm -rf /tmp/ep03-terraform-demo /tmp/ep03-bucket
```

This removes the k3d cluster and the temporary Terraform state directories.
