# Multi-Cluster GitOps Without the Pain

Here's a problem nobody warns you about with Argo CD at scale.

The traditional hub-spoke setup has your control plane **reaching OUT** to every cluster it manages. That means open firewall rules, exposed API servers, and stored kubeconfig credentials for every single workload cluster. Add fifty clusters and you've got fifty attack surfaces baked directly into your control plane.

**argocd-agent flips this completely.** Instead of the hub reaching out — the agents reach in. Each workload cluster runs a lightweight agent that connects back to the hub over gRPC with mutual TLS. Your control plane never needs to touch your clusters directly. Ever.

This post covers the full homelab setup: k3d hub cluster, two vClusters as simulated workload targets, PKI bootstrapping, and a live GitOps application sync — all from a single script you can run and tear down in under 30 minutes.

## Architecture Overlay

### Before: Traditional Argo CD Hub-Spoke (Push Model)

In a standard Argo CD multi-cluster setup, the hub stores a kubeconfig secret for every workload cluster. The `application-controller` pushes manifests **outward** into each cluster over the Kubernetes API. Every workload cluster must expose its API server to the hub.

```
┌─────────────────────────────────────────────────────────────┐
│                     Hub Cluster (Argo CD)                   │
│                                                             │
│   argocd-server          application-controller             │
│                                                             │
│   Secrets:                                                  │
│     kubeconfig-workload-1  ──────────────────────────────┐  │
│     kubeconfig-workload-2  ───────────────────────────┐  │  │
│     kubeconfig-workload-N  ──────────────────────┐    │  │  │
└──────────────────────────────────────────────────┼────┼──┼──┘
                                                   │    │  │
                        outbound push ─────────────▼────▼──▼
                                      (hub reaches INTO each cluster)
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│   Workload-1     │  │   Workload-2     │  │   Workload-N     │
│                  │  │                  │  │                  │
│  ⚠ API server    │  │  ⚠ API server    │  │  ⚠ API server    │
│    exposed       │  │    exposed       │  │    exposed       │
└──────────────────┘  └──────────────────┘  └──────────────────┘

  Each cluster must be reachable from the hub.
  Each kubeconfig secret = credential that, if leaked, owns that cluster.
```

### After: argocd-agent Pull Model (Egress-Only)

argocd-agent reverses the direction. Each workload cluster runs a lightweight agent that **dials out** to the `argocd-agent-principal` on the hub over gRPC + mTLS. The hub stores no kubeconfig secrets and makes zero inbound connections to workload clusters.

```
┌─────────────────────────────────────────────────────────────┐
│                     Hub Cluster                             │
│                                                             │
│   argocd-server          argocd-agent-principal             │
│                              (gRPC server :8443)            │
│   application-controller ← scaled to 0 on hub              │
│                                                             │
│   No kubeconfig secrets. No outbound connections.           │
└──────────────────────────┬──────────────────────────────────┘
                           │ ▲
                           │ │  agents dial OUT over gRPC + mTLS
                           │ │  (hub never initiates)
              ┌────────────┘ └────────────┐
              │                           │
┌─────────────▼────────────┐  ┌───────────▼──────────────┐
│       Workload-1         │  │       Workload-2          │
│                          │  │                           │
│  argocd-agent-agent      │  │  argocd-agent-agent       │
│  application-controller  │  │  application-controller   │
│                          │  │                           │
│  ✔ no inbound required   │  │  ✔ no inbound required    │
│  ✔ egress-only           │  │  ✔ egress-only            │
└──────────────────────────┘  └───────────────────────────┘

  Works behind NAT, in air-gapped environments, at the edge.
  Namespace workload-1 on hub → routes to Workload-1 agent via gRPC.
```

**What changes operationally:**
- Workload clusters need **zero inbound firewall rules** — outbound port 8443 is enough
- No kubeconfig secrets on the hub — nothing to rotate, nothing to leak
- Each agent authenticates with a unique mTLS client certificate issued by your own CA
- The hub's `application-controller` is scaled to 0; each workload runs its own

## Step-by-Step Implementation

### Step 1 — Create the Hub Cluster

The hub runs Argo CD plus the principal component. k3d exposes port 8443 for the Argo CD UI.

```bash
k3d cluster create hub \
  --port "8080:80@loadbalancer" \
  --port "8443:443@loadbalancer" \
  --agents 1 \
  --wait

kubectl config use-context k3d-hub
```

### Step 2 — Create vClusters as Workload Targets

vCluster gives you isolated Kubernetes API servers running inside the k3d node. Each vCluster simulates a remote workload cluster without needing separate physical machines.

```bash
# Create workload-1
vcluster create workload-1 \
  --namespace vcluster-workload-1 \
  --context k3d-hub \
  --connect=false

# Merge vCluster kubeconfig into ~/.kube/config
vcluster connect workload-1 \
  -n vcluster-workload-1 \
  --context k3d-hub \
  --update-current=false \
  --kube-config-context-name vcluster-workload-1 \
  --kube-config /tmp/vc-workload-1.yaml

KUBECONFIG="${HOME}/.kube/config:/tmp/vc-workload-1.yaml" \
  kubectl config view --flatten > /tmp/merged.yaml && \
  mv /tmp/merged.yaml "${HOME}/.kube/config"

# Repeat for workload-2
vcluster create workload-2 \
  --namespace vcluster-workload-2 \
  --context k3d-hub \
  --connect=false

vcluster connect workload-2 \
  -n vcluster-workload-2 \
  --context k3d-hub \
  --update-current=false \
  --kube-config-context-name vcluster-workload-2 \
  --kube-config /tmp/vc-workload-2.yaml

KUBECONFIG="${HOME}/.kube/config:/tmp/vc-workload-2.yaml" \
  kubectl config view --flatten > /tmp/merged.yaml && \
  mv /tmp/merged.yaml "${HOME}/.kube/config"
```

Verify all three contexts are present:

```bash
kubectl config get-contexts
# Expected: k3d-hub, vcluster-workload-1, vcluster-workload-2
```

### Step 3 — Install Argo CD on the Hub

Install Argo CD, then **scale the application-controller to zero replicas**. This is the critical architectural change — reconciliation will happen on workload clusters instead.

```bash
kubectl create namespace argocd --context k3d-hub 2>/dev/null || true

kubectl apply -n argocd \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml \
  --context k3d-hub \
  --server-side --force-conflicts

# Wait for argocd-server to be ready
kubectl rollout status deployment/argocd-server \
  -n argocd --context k3d-hub --timeout=300s

# Scale app-controller to 0 on hub — controllers run on workload clusters
# Note: argocd-application-controller is a StatefulSet in Argo CD v3+
kubectl scale statefulset argocd-application-controller \
  -n argocd --replicas=0 --context k3d-hub

# Save admin password for later
ARGOCD_PASS=$(kubectl get secret argocd-initial-admin-secret \
  -n argocd --context k3d-hub \
  -o jsonpath="{.data.password}" | base64 -d)
echo "${ARGOCD_PASS}" > /tmp/argocd-admin-password.txt
echo "Admin password: ${ARGOCD_PASS}"
```

### Step 4 — Bootstrap the PKI

argocd-agentctl manages the certificate authority and all mTLS certificates.

```bash
# Initialize the CA
argocd-agentctl pki init \
  --principal-context k3d-hub \
  --principal-namespace argocd

# Install the principal component
kubectl apply -n argocd \
  -k "https://github.com/argoproj-labs/argocd-agent/install/kubernetes/principal?ref=main" \
  --context k3d-hub

# Get principal service ClusterIP and k3d node IP for cert SANs
# (The principal is a LoadBalancer service; agents reach it via NodePort from vClusters)
PRINCIPAL_IP=$(kubectl get svc argocd-agent-principal \
  -n argocd --context k3d-hub \
  -o jsonpath='{.spec.clusterIP}')
NODE_IP=$(kubectl get nodes --context k3d-hub \
  -o jsonpath='{.items[0].status.addresses[?(@.type=="InternalIP")].address}')

# Issue principal TLS certificate — include BOTH ClusterIP and node IP in SANs
# The cert must be issued before the principal can start (it crashes without it)
argocd-agentctl pki issue principal \
  --principal-context k3d-hub \
  --ip "${PRINCIPAL_IP}" \
  --ip "${NODE_IP}" \
  --dns "argocd-agent-principal.argocd.svc.cluster.local" \
  --upsert

# Issue resource-proxy certificate
argocd-agentctl pki issue resource-proxy \
  --principal-context k3d-hub \
  --dns "argocd-agent-resource-proxy.argocd.svc.cluster.local" \
  --upsert

# Create JWT signing key
argocd-agentctl jwt create-key \
  --principal-context k3d-hub \
  --principal-namespace argocd \
  --upsert

# Now wait for principal to be ready (certs are in place so it will start successfully)
kubectl rollout status deployment/argocd-agent-principal \
  -n argocd --context k3d-hub --timeout=300s

# Restart principal to apply all certs cleanly
kubectl rollout restart deployment/argocd-agent-principal \
  -n argocd --context k3d-hub

kubectl rollout status deployment/argocd-agent-principal \
  -n argocd --context k3d-hub --timeout=300s
```

### Step 4.5 — Configure the Principal (Namespace Routing + Redis Access)

This step is required for k3d environments and for namespace-based agent routing to work. Without it, agents connect but Applications never route to the correct workload cluster.

```bash
# Patch Redis NetworkPolicy to allow the principal pod access
kubectl patch networkpolicy argocd-redis-network-policy \
  -n argocd --context k3d-hub \
  --type=json \
  -p='[{"op":"add","path":"/spec/ingress/0/from/-","value":{"podSelector":{"matchLabels":{"app.kubernetes.io/name":"argocd-agent-principal"}}}}]' 2>/dev/null || true

# Pin principal to the same node as argocd-redis (k3d cross-node ClusterIP issue)
REDIS_NODE=$(kubectl get pod -n argocd -l "app.kubernetes.io/name=argocd-redis" \
  --context k3d-hub -o jsonpath='{.items[0].spec.nodeName}')
kubectl patch deployment argocd-agent-principal -n argocd --context k3d-hub \
  -p="{\"spec\":{\"template\":{\"spec\":{\"nodeSelector\":{\"kubernetes.io/hostname\":\"${REDIS_NODE}\"}}}}}"

# Enable namespace-based routing: Application namespace = agent name
kubectl patch configmap argocd-agent-params \
  -n argocd --context k3d-hub \
  --type=merge \
  -p='{"data":{"principal.allowed-namespaces":"workload-1,workload-2"}}'

# Restart principal to pick up all config changes
kubectl rollout restart deployment/argocd-agent-principal -n argocd --context k3d-hub
kubectl rollout status deployment/argocd-agent-principal \
  -n argocd --context k3d-hub --timeout=300s
```

**Why this matters:** `principal.allowed-namespaces` tells the principal which Kubernetes namespaces map to registered agent names. An Application in namespace `workload-1` on the hub will be routed to the `workload-1` agent. Without this config, the routing table is empty and no applications are delivered.

### Step 5 — Register Agents and Issue mTLS Client Certificates

Each workload cluster gets its own registered agent identity and a unique mTLS client certificate.

```bash
# Pre-create the argocd namespace on workload clusters before issuing certs
for agent in workload-1 workload-2; do
  kubectl create namespace argocd --context "vcluster-${agent}" 2>/dev/null || true
done

for agent in workload-1 workload-2; do
  # Register the agent (port is included in --resource-proxy-server address)
  argocd-agentctl agent create "${agent}" \
    --principal-context k3d-hub \
    --principal-namespace argocd \
    --resource-proxy-server "argocd-agent-resource-proxy.argocd.svc.cluster.local:9090"

  # Issue a unique mTLS client cert for this agent
  argocd-agentctl pki issue agent "${agent}" \
    --principal-context k3d-hub \
    --agent-context "vcluster-${agent}" \
    --agent-namespace argocd \
    --upsert

  # Propagate CA cert to the agent cluster (agents need this to verify the principal)
  argocd-agentctl pki propagate \
    --principal-context k3d-hub \
    --principal-namespace argocd \
    --agent-context "vcluster-${agent}" \
    --agent-namespace argocd
done
```

### Step 5.5 — Register Cluster Secrets

Argo CD needs a cluster secret for each workload before it can route Applications to them. These secrets tell Argo CD to reach the workload clusters **through the resource proxy** — not directly.

```bash
PROXY_ADDR="argocd-agent-resource-proxy.argocd.svc.cluster.local"

for agent in workload-1 workload-2; do
  kubectl apply -n argocd --context k3d-hub -f - <<EOF
apiVersion: v1
kind: Secret
metadata:
  name: cluster-${agent}
  namespace: argocd
  labels:
    argocd.argoproj.io/secret-type: cluster
    argocd-agent.argoproj-labs.io/agent-name: ${agent}
stringData:
  name: "${agent}"
  server: "https://${PROXY_ADDR}:9090?agentName=${agent}"
  config: '{"tlsClientConfig":{"insecure":true}}'
type: Opaque
EOF
done
```

**Why this matters:** The `argocd.argoproj.io/secret-type: cluster` label registers the workload as a known cluster destination in Argo CD. The `server` URL points to the resource proxy — a sidecar on the principal that translates Argo CD's cluster API calls into agent-routed gRPC messages. Without this secret, Argo CD has nowhere to send the Application.

### Step 6 — Install Agent + App-Controller on Workload Clusters

Each workload cluster gets the agent and its own application-controller. The agent is configured with the principal's address.

```bash
# vCluster pods cannot resolve host cluster services via DNS.
# Agents must connect using the k3d node IP + the principal service's NodePort.
PRINCIPAL_NODE_IP=$(kubectl get nodes --context k3d-hub \
  -o jsonpath='{.items[0].status.addresses[?(@.type=="InternalIP")].address}')
PRINCIPAL_NODE_PORT=$(kubectl get svc argocd-agent-principal \
  -n argocd --context k3d-hub \
  -o jsonpath='{.spec.ports[0].nodePort}')

echo "Principal reachable from vClusters at: ${PRINCIPAL_NODE_IP}:${PRINCIPAL_NODE_PORT}"

for ctx in vcluster-workload-1 vcluster-workload-2; do
  kubectl create namespace argocd --context "${ctx}" 2>/dev/null || true

  # Install Argo CD (for the application-controller only)
  kubectl apply -n argocd \
    -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml \
    --context "${ctx}" \
    --server-side --force-conflicts

  kubectl rollout status statefulset/argocd-application-controller \
    -n argocd --context "${ctx}" --timeout=300s

  # Scale down everything except app-controller and repo-server on workloads
  for d in argocd-server argocd-applicationset-controller \
            argocd-notifications-controller argocd-dex-server; do
    kubectl scale deployment "${d}" \
      -n argocd --replicas=0 --context "${ctx}" 2>/dev/null || true
  done

  # Install the agent
  kubectl apply -n argocd \
    -k "https://github.com/argoproj-labs/argocd-agent/install/kubernetes/agent?ref=main" \
    --context "${ctx}"

  # Configure agent: use node IP + NodePort (not DNS — vClusters can't resolve host services)
  kubectl patch configmap argocd-agent-params \
    -n argocd --context "${ctx}" \
    --patch "{
      \"data\": {
        \"agent.server.address\": \"${PRINCIPAL_NODE_IP}\",
        \"agent.server.port\": \"${PRINCIPAL_NODE_PORT}\",
        \"agent.mode\": \"managed\",
        \"agent.creds\": \"mtls:^CN=(.+)$\",
        \"agent.tls.secret-name\": \"argocd-agent-client-tls\",
        \"agent.tls.client.insecure\": \"false\"
      }
    }"

  kubectl rollout restart deployment/argocd-agent-agent \
    -n argocd --context "${ctx}"

  kubectl rollout status deployment/argocd-agent-agent \
    -n argocd --context "${ctx}" --timeout=300s
done

# Create routing namespaces on hub — Applications go here to be routed to the matching agent
kubectl create namespace workload-1 --context k3d-hub 2>/dev/null || true
kubectl create namespace workload-2 --context k3d-hub 2>/dev/null || true
```

### Step 7 — Start the Argo CD UI

Port-forward the Argo CD server from the hub:

```bash
kubectl port-forward svc/argocd-server \
  -n argocd 8443:443 \
  --context k3d-hub \
  &>/tmp/argocd-port-forward.log &

sleep 2
echo "UI: https://localhost:8443"
echo "Login: admin / $(cat /tmp/argocd-admin-password.txt)"
```

## POC / Verification

### Verify Agents Connected

Check agent logs for the connection handshake. Give the agents 15 seconds to establish their gRPC connection.

```bash
sleep 15

# Check agent on workload-1 — look for connection/register lines
kubectl logs deployment/argocd-agent-agent \
  -n argocd --context vcluster-workload-1 \
  | grep -iE "connect|register|established" | tail -8

# Check agent on workload-2
kubectl logs deployment/argocd-agent-agent \
  -n argocd --context vcluster-workload-2 \
  | grep -iE "connect|register|established" | tail -8

# Check principal — should show both agents registered
kubectl logs deployment/argocd-agent-principal \
  -n argocd --context k3d-hub \
  | grep -iE "connect|agent|register" | tail -8
```

**Expected output on the agent:** a log line showing the gRPC connection established to the principal over mTLS. **Expected output on the principal:** log lines showing both `workload-1` and `workload-2` agents registered.

### Deploy a Test Application

Apply an Application manifest on the **hub**. The agent relays it to the workload cluster via gRPC, and the workload's own app-controller applies it locally.

```yaml
# manifests/demo-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: guestbook
  namespace: workload-1
spec:
  project: default
  source:
    repoURL: https://github.com/argoproj/argocd-example-apps
    targetRevision: HEAD
    path: guestbook
  destination:
    server: https://kubernetes.default.svc
    namespace: guestbook
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
```

```bash
kubectl apply -f manifests/demo-app.yaml --context k3d-hub

# Watch sync propagate — check status on the hub
sleep 30
kubectl get application guestbook -n workload-1 --context k3d-hub

# Verify pods deployed on workload-1
kubectl get pods -n guestbook --context vcluster-workload-1
```

The application should appear as `Synced` and `Healthy` on the workload cluster — without the hub ever having touched the workload cluster's API server directly.

### Troubleshooting: Agents Won't Connect (TLS Errors)

If agent logs show TLS handshake errors after setup, the principal's certificate may not include the current ClusterIP in its SAN. Re-issue it:

```bash
./setup.sh --fix-tls
# Re-issues the principal cert with the current ClusterIP and restarts the principal
# Agents should reconnect within 30 seconds
```

### Expected: `dial tcp` connection errors in agent logs

When watching agent logs during setup, you may see intermittent lines like:

```
dial tcp 10.43.x.x:443: connect: connection refused
```

These are **expected and self-healing**. The vCluster API server briefly rejects connections while its internal components are initializing. The agent retries automatically and the errors stop within 30–60 seconds of the vCluster becoming fully healthy. They do not indicate a misconfiguration.

### Teardown

When you're done, the teardown is fully automated:

```bash
./setup.sh --teardown
# Prompts for confirmation, then removes clusters, contexts, and /tmp files
```

## What You Built

You now have a working argocd-agent environment where:

- **Workload clusters initiate all connections** — zero inbound access required
- **mTLS mutual authentication** replaces stored kubeconfig credentials
- **app-controllers run on workload clusters** — reconciliation happens at the edge
- **The hub is purely a control plane** — it defines applications, the agents deliver them

This architecture scales to hundreds of workload clusters without adding more attack surface to your hub. It works in air-gapped environments and behind NAT where traditional Argo CD would require VPN tunnels or jump hosts.
