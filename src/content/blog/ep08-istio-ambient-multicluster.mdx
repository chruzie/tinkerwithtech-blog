Every Istio sidecar injects roughly 50 MB of Envoy into your pod. Multiply that across a large fleet and you're paying real memory budget for proxy overhead before your application runs a single request. Sidecar misconfiguration also takes the pod with it — a wrong `DestinationRule` can put your service in a crash loop.

Ambient mode changes the resource model. ztunnel runs as a DaemonSet — one proxy per node, shared by every pod on that node. It handles L4 mTLS, SPIFFE identity, and basic routing. When you need L7 — HTTP-aware `AuthorizationPolicy`, header matching, gRPC routing — you add a waypoint proxy per namespace. The per-pod Envoy sidecar is gone from both paths.

## Architecture

The setup uses two k3d clusters sharing Docker networking. `k3d-mesh-primary` runs the primary Istio control plane (`istiod`). `k3d-mesh-remote` runs without a control plane — it discovers workloads and receives mesh config entirely from the primary's istiod via the east-west gateway.

```
k3d-mesh-primary                          k3d-mesh-remote
────────────────────────────────          ────────────────────────────────
istio-system                              istio-system
  istiod           (control plane)          ztunnel DaemonSet
  ztunnel DaemonSet                         east-west GW (port 15443)
  east-west GW (port 15443)
  Kiali + Prometheus                      httpbin namespace  [ambient]
                                            httpbin deployment
                                            waypoint proxy
                                          default namespace  [ambient]
                                            curl pod
                                            ServiceAccount: curl

HERO MOMENT request path (intra-cluster, remote):
  curl → ztunnel@remote (L4 mTLS via HBONE)
       → waypoint proxy (L7 AuthorizationPolicy enforcement)
       → httpbin

SPIFFE identity on the wire:
  caller:      spiffe://cluster.local/ns/default/sa/curl
  destination: spiffe://cluster.local/ns/httpbin/sa/default
  cert issuer: istiod on primary (single trust domain for both clusters)
```

> **k3d networking note:** k3d cluster pods are isolated in separate CIDRs (e.g., `10.42.x.x` on primary, `10.43.x.x` on remote). These CIDRs are not routable between clusters — only the Docker bridge network (`172.18.x.x`) connects cluster nodes. Istio ambient's ztunnel workload discovery (the `Address` xDS stream) uses pod IPs to build the workload table, so cross-cluster pod-to-pod HBONE tunnels cannot form in k3d without custom routing rules. The east-west gateway is still deployed and operational for control plane connectivity (istiod endpoint discovery, CA bundle sync) — but the HERO MOMENT demo runs on the remote cluster where ztunnel + waypoint + curl are all co-located. The ambient concepts (sidecar-free mTLS, L7 AuthorizationPolicy via waypoint) are identical regardless of whether the call crosses clusters.

### ztunnel vs waypoint — when each applies

ztunnel handles L4: encryption, SPIFFE identity verification, and HBONE tunnel establishment. Every pod in an ambient-enrolled namespace gets ztunnel coverage automatically at the node level. No annotation required on the pod.

The waypoint proxy is optional and per-namespace. Add it when you need HTTP-aware policy — `AuthorizationPolicy` with `methods`, `paths`, or `headers` fields. Without a waypoint, `AuthorizationPolicy` still applies at L4 (by source principal) but cannot inspect HTTP semantics.

In this episode `httpbin` gets a waypoint on the remote cluster. The `curl` client also runs on the remote cluster in the `default` namespace — both are enrolled in ambient mode there.

### Multi-cluster single mesh

Both clusters share one trust domain (`cluster.local`), one `istiod`, and one certificate authority. `istioctl create-remote-secret` gives `istiod` on the primary read access to the remote cluster's Kubernetes API server, enabling endpoint discovery across clusters. The east-west gateway on each cluster handles cross-cluster packet forwarding on port 15443 using TLS `AUTO_PASSTHROUGH` — ztunnel owns the encryption, the gateway just routes.

This is a single-mesh setup, not multi-mesh. SPIFFE identities from both clusters are trusted without cross-mesh federation configuration.

## Step-by-step

<Step number={1} title="Create two k3d clusters on a shared Docker network">

k3d maps each cluster's API server to a host port (`0.0.0.0:6443`, `0.0.0.0:6444`). Without a shared Docker network, `istioctl create-remote-secret` generates a server URL of `https://0.0.0.0:6444` — unreachable from inside Docker. Placing both clusters on the same Docker bridge network lets them reach each other via container IPs instead.

Create the network first:

```bash
docker network create k3d-mesh-network
```

Then create both clusters with `--network k3d-mesh-network`. Both clusters also disable Flannel so Cilium can take over as CNI. Traefik and the built-in service load balancer are disabled:

```bash
k3d cluster create mesh-primary \
  --network k3d-mesh-network \
  --api-port 6443 \
  --k3s-arg "--disable=traefik@server:0" \
  --k3s-arg "--disable=servicelb@server:0" \
  --k3s-arg "--flannel-backend=none@server:0" \
  --wait

k3d cluster create mesh-remote \
  --network k3d-mesh-network \
  --api-port 6444 \
  --k3s-arg "--disable=traefik@server:0" \
  --k3s-arg "--disable=servicelb@server:0" \
  --k3s-arg "--flannel-backend=none@server:0" \
  --wait
```

Verify both cluster containers are on the shared network:

```bash
docker network inspect k3d-mesh-network \
  --format '{{range .Containers}}{{.Name}} {{end}}'
```

Then switch to the primary context:

```bash
kubectl config use-context k3d-mesh-primary
```

</Step>

<Step number={2} title="Install Cilium CNI on both clusters">

Istio ambient mode requires a CNI that supports the node-level network interception ztunnel depends on. Cilium with `kubeProxyReplacement=false` is the supported configuration for k3d:

```bash
helm repo add cilium https://helm.cilium.io/ --force-update

for ctx in k3d-mesh-primary k3d-mesh-remote; do
  helm install cilium cilium/cilium \
    --version 1.15.5 \
    --namespace kube-system \
    --kube-context "${ctx}" \
    --set operator.replicas=1 \
    --set ipam.mode=kubernetes \
    --set kubeProxyReplacement=false \
    --wait \
    --timeout 5m
done
```

Wait for nodes to be Ready on both clusters before proceeding — ztunnel will crash if Cilium hasn't finished initializing:

```bash
for ctx in k3d-mesh-primary k3d-mesh-remote; do
  kubectl wait node --all --for=condition=Ready \
    --context "${ctx}" --timeout=120s
done
```

</Step>

<Step number={3} title="Install Istio ambient on the primary cluster">

The repo includes an umbrella Helm chart that bundles all four Istio ambient components — base CRDs, `istiod`, Istio CNI, and ztunnel. The install must happen in three phases because ztunnel's DaemonSet requires the CNI socket (`/var/run/ztunnel/ztunnel.sock`) to pass its readiness probe — if ztunnel starts before CNI is ready, Helm's `--wait` will time out in a deadlock.

**Phase 1** — install Istio CNI (creates the namespace and the socket ztunnel needs):

```bash
helm install istio-cni istio/cni \
  --namespace istio-system \
  --create-namespace \
  --kube-context k3d-mesh-primary \
  --set profile=ambient \
  --wait --timeout 5m
```

**Phase 2** — install istiod and base CRDs, with ztunnel disabled:

```bash
helm install istio-ambient ./charts/istio-ambient \
  --namespace istio-system \
  --kube-context k3d-mesh-primary \
  -f ./charts/istio-ambient/values-primary.yaml \
  --set ztunnel.enabled=false \
  --wait --timeout 5m
```

**Phase 3** — delete the validating webhook (Helm v4 SSA conflict on upgrade) and enable ztunnel:

```bash
kubectl delete validatingwebhookconfiguration \
  istiod-default-validator istio-validator-istio-system \
  --context k3d-mesh-primary 2>/dev/null || true

helm upgrade istio-ambient ./charts/istio-ambient \
  --namespace istio-system \
  --kube-context k3d-mesh-primary \
  -f ./charts/istio-ambient/values-primary.yaml \
  --wait --timeout 5m
```

The `values-primary.yaml` file sets `istiod.enabled: true` and `global.multiCluster.clusterName: mesh-primary`. The cluster name is required — without it istiod only knows about `"Kubernetes"` and ztunnel fails with `"client claims to be in cluster 'mesh-primary', but we only know about local cluster 'Kubernetes'"`.

The `setup.sh` script handles all three phases automatically. The webhook deletion step is idempotent — it uses `|| true` so re-runs don't fail if the webhook has already been removed.

Confirm the ztunnel DaemonSet rolled out:

```bash
kubectl rollout status daemonset/ztunnel \
  -n istio-system --context k3d-mesh-primary --timeout=180s
```

</Step>

<Step number={4} title="Join the remote cluster to the mesh">

The remote cluster uses the same umbrella chart with a local istiod for injection and CA proxying — the primary's istiod remains the mesh config source of truth. The install is two phases: first install without ztunnel, then add ztunnel after the east-west gateway establishes cross-cluster connectivity.

**Phase 1** — install base CRDs and Istio CNI only (ztunnel disabled):

```bash
helm install istio-ambient ./charts/istio-ambient \
  --namespace istio-system \
  --create-namespace \
  --kube-context k3d-mesh-remote \
  -f ./charts/istio-ambient/values-remote.yaml \
  --set ztunnel.enabled=false \
  --wait
```

Install the east-west gateways on both clusters. The manifests use Istio auto-injection (`inject.istio.io/templates: gateway`), so enable injection on `istio-system` first — this is what provides the `/var/run/secrets/istio/root-cert.pem` volume mount the gateway pod needs:

```bash
kubectl label namespace istio-system istio-injection=enabled \
  --context=k3d-mesh-primary --overwrite

kubectl label namespace istio-system istio-injection=enabled \
  --context=k3d-mesh-remote --overwrite

kubectl apply --context k3d-mesh-primary \
  -f manifests/east-west-gateway.yaml

kubectl apply --context k3d-mesh-remote \
  -f manifests/east-west-gateway.yaml
```

After the gateway pods are `Ready`, expose istiod via a `ServiceEntry` pointing at each gateway's pod IP. k3d has no LoadBalancer controller, so the Service `EXTERNAL-IP` stays `<pending>` — using the pod IP directly is the reliable approach:

```bash
for CTX in k3d-mesh-primary k3d-mesh-remote; do
  EW_POD_IP=$(kubectl get pod \
    -l app=istio-eastwestgateway \
    -n istio-system \
    --context="${CTX}" \
    -o jsonpath='{.items[0].status.podIP}')

  kubectl apply --context="${CTX}" -f - <<EOF
apiVersion: networking.istio.io/v1alpha3
kind: Gateway
metadata:
  name: cross-network-gateway
  namespace: istio-system
spec:
  selector:
    istio: eastwestgateway
  servers:
  - port:
      number: 15443
      name: tls
      protocol: TLS
    tls:
      mode: AUTO_PASSTHROUGH
    hosts:
    - "*.local"
---
apiVersion: networking.istio.io/v1alpha3
kind: ServiceEntry
metadata:
  name: istiod-mesh-primary
  namespace: istio-system
spec:
  hosts:
  - istiod.istio-system.svc.cluster.local
  ports:
  - number: 15012
    name: tls-istiod
    protocol: TLS
  resolution: STATIC
  endpoints:
  - address: "${EW_POD_IP}"
    ports:
      tls-istiod: 15012
    labels:
      app: istiod
EOF
done
```

Grant `istiod` full read access to the remote cluster before creating the remote secret. Without this, istiod gets a 403 Forbidden when listing namespaces during endpoint discovery. The `view` ClusterRole is insufficient — istiod also needs to read Secrets and ServiceAccounts for SPIFFE cert issuance, which `view` omits:

```bash
kubectl create clusterrolebinding istio-reader-binding \
  --clusterrole=cluster-admin \
  --serviceaccount=istio-system:istio-reader-service-account \
  --context=k3d-mesh-remote
```

Create the remote secret using the Docker-internal container IP, not the host-mapped `0.0.0.0` port. The `--server` flag overrides what `istioctl` would otherwise embed:

```bash
# Resolve the Docker-internal IP of the remote API server container
REMOTE_API_IP=$(docker inspect k3d-mesh-remote-server-0 \
  --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' | head -1)

echo "Remote API server container IP: ${REMOTE_API_IP}"

istioctl create-remote-secret \
  --context=k3d-mesh-remote \
  --name=mesh-remote \
  --server="https://${REMOTE_API_IP}:6443" \
  | kubectl apply --context=k3d-mesh-primary -f -
```

> **Why not `0.0.0.0:6444`?** k3d exposes each cluster's API server via a host port binding. When `istioctl create-remote-secret` omits `--server`, it picks up `https://0.0.0.0:6444` from the kubeconfig. That address is the host machine's loopback — unreachable from inside the `istiod` container. Using the Docker-internal container IP means istiod can reach the remote API server directly over the shared Docker bridge network.

Copy the CA root cert from primary to remote, then sync the trust bundle before starting ztunnel:

```bash
kubectl get configmap istio-ca-root-cert \
  -n istio-system \
  --context=k3d-mesh-primary \
  -o yaml \
  | kubectl apply --context=k3d-mesh-remote -n istio-system -f -
```

**Phase 2** — upgrade the release to enable ztunnel now that istiod is reachable via the east-west gateway:

```bash
helm upgrade istio-ambient ./charts/istio-ambient \
  --namespace istio-system \
  --kube-context k3d-mesh-remote \
  -f ./charts/istio-ambient/values-remote.yaml \
  --wait
```

The `values-remote.yaml` file sets `ztunnel.enabled: true` and `multiCluster.clusterName: mesh-remote`. The cluster name must exactly match the `--name` flag passed to `create-remote-secret`. A mismatch causes endpoint discovery failures that are difficult to diagnose from logs alone.

</Step>

<Step number={5} title="East-west gateway reference">

The east-west gateway manifest applied in Step 4 runs Envoy in `AUTO_PASSTHROUGH` mode on port 15443 — it forwards raw TLS without terminating it, so ztunnel remains responsible for encryption.

The gateway manifest includes a `Gateway` resource with `tls.mode: AUTO_PASSTHROUGH` on port 15443 and a `ServiceEntry` that exposes istiod across the mesh:

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: Gateway
metadata:
  name: cross-network-gateway
  namespace: istio-system
spec:
  selector:
    istio: eastwestgateway
  servers:
  - port:
      number: 15443
      name: tls
      protocol: TLS
    tls:
      mode: AUTO_PASSTHROUGH
    hosts:
    - "*.local"
```

The east-west gateway Service type is `NodePort`. k3d has no cloud LoadBalancer controller so `LoadBalancer` services stay permanently `<pending>`. The setup script uses the pod IP directly in the `ServiceEntry` endpoint, so no external IP is needed. NodePort provides the listening socket ztunnel needs on port 15443 for the control-plane east-west channel.

</Step>

<Step number={6} title="Deploy httpbin and curl client">

Create the `httpbin` namespace on the remote cluster and enroll it in the ambient mesh with a single label:

```bash
kubectl create namespace httpbin --context k3d-mesh-remote

kubectl label namespace httpbin \
  istio.io/dataplane-mode=ambient \
  --context k3d-mesh-remote --overwrite
```

Deploy httpbin:

```bash
kubectl apply --context k3d-mesh-remote \
  -n httpbin \
  -f manifests/httpbin.yaml
```

The manifest uses `docker.io/kong/httpbin:0.1.0` — a maintained fork of the original `kennethreitz/httpbin`. Enroll the `default` namespace on the **remote** cluster and deploy the curl client there. The curl client and httpbin are co-located on the remote cluster so that both are intercepted by the same ztunnel DaemonSet — this is what enables the sidecar-free mTLS path and waypoint L7 enforcement in k3d (see architecture note above):

```bash
kubectl label namespace default \
  istio.io/dataplane-mode=ambient \
  --context k3d-mesh-remote --overwrite

kubectl apply --context k3d-mesh-remote \
  -f manifests/curl-client.yaml
```

The curl client manifest creates a `ServiceAccount` named `curl` and a pod that uses it. The service account name becomes part of the SPIFFE identity:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: curl
  namespace: default
---
apiVersion: v1
kind: Pod
metadata:
  name: curl
  namespace: default
spec:
  serviceAccountName: curl
  containers:
  - name: curl
    image: curlimages/curl:8.4.0
    command: ["sleep", "infinity"]
```

Wait for the curl pod:

```bash
kubectl wait pod curl \
  --for=condition=Ready \
  --context k3d-mesh-remote \
  --timeout=120s
```

</Step>

<Step number={7} title="Install Kiali and Prometheus">

Kiali needs Prometheus for the mesh graph. Install both on the primary cluster:

```bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts --force-update
helm repo add kiali https://kiali.org/helm-charts --force-update
helm repo update

helm install prometheus prometheus-community/prometheus \
  --namespace istio-system \
  --kube-context k3d-mesh-primary \
  --set alertmanager.enabled=false \
  --set pushgateway.enabled=false \
  --set server.persistentVolume.enabled=false \
  --wait \
  --timeout 5m

helm install kiali-server kiali/kiali-server \
  --namespace istio-system \
  --kube-context k3d-mesh-primary \
  --set auth.strategy=anonymous \
  --set external_services.prometheus.url="http://prometheus-server.istio-system.svc.cluster.local" \
  --set-string "deployment.pod_annotations.sidecar\.istio\.io/inject=false" \
  --wait \
  --timeout 5m
```

The `sidecar.istio.io/inject=false` annotation is required here. The `istio-system` namespace has `istio-injection=enabled` (applied for east-west gateway auto-injection), which causes Kiali to receive an `istio-proxy` native sidecar init container. That init container fails its readiness probe because ztunnel's health endpoint is not exposed on port 15021. Disabling injection on the Kiali pod prevents this.

Kiali needs traffic before the mesh graph populates. Run 10–20 GET requests after the full setup and wait at least 2 minutes for Prometheus to scrape the metrics. Prometheus scrape interval defaults to 30s.

</Step>

<Step number={8} title="Add waypoint proxy and AuthorizationPolicy">

The waypoint proxy adds L7 enforcement to the `httpbin` namespace. `--enroll-namespace` sets the `istio.io/use-waypoint: waypoint` annotation on the namespace — without that annotation, L7 policies are not enforced even when the waypoint pod is running.

```bash
istioctl waypoint apply \
  --context=k3d-mesh-remote \
  --namespace httpbin \
  --enroll-namespace \
  --wait
```

Apply the `AuthorizationPolicy`:

```bash
kubectl apply --context k3d-mesh-remote \
  -f manifests/authz-policy.yaml
```

The policy targets the `httpbin` Service by `targetRefs` (the waypoint-aware selector) and allows only GET requests from any caller:

```yaml
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: httpbin-allow-get-only
  namespace: httpbin
spec:
  targetRefs:
  - kind: Service
    group: ""
    name: httpbin
  rules:
  - to:
    - operation:
        methods:
        - GET
```

The key demo point is HTTP method enforcement via the waypoint proxy — GET passes through, POST returns `403 RBAC: access denied`. Adding a `from.source.principals` field would also restrict by SPIFFE identity (the principal format is `cluster.local/ns/<namespace>/sa/<serviceaccount>`, without the `spiffe://` prefix). That field is optional here — the method enforcement alone demonstrates L7-aware policy at the waypoint.

</Step>

<Step number={9} title="Hero — GET allowed, POST blocked, Kiali confirms">

Send a GET request from curl to httpbin on the remote cluster. Both pods are enrolled in ambient mode, so ztunnel intercepts the traffic and wraps it in HBONE mTLS — zero sidecars:

```bash
kubectl exec pod/curl --context k3d-mesh-remote -- \
  curl -sv "http://httpbin.httpbin.svc.cluster.local/get" 2>&1 | \
  grep -E "< HTTP|origin|X-Forwarded"
```

Expected output: `< HTTP/1.1 200 OK` and a JSON body from httpbin.

Send a POST request — the waypoint blocks it at L7:

```bash
kubectl exec pod/curl --context k3d-mesh-remote -- \
  curl -sv -X POST "http://httpbin.httpbin.svc.cluster.local/post" 2>&1 | \
  grep -E "< HTTP|RBAC|access denied|403"
```

Expected output: `< HTTP/1.1 403 Forbidden` with body `RBAC: access denied`. The waypoint proxy intercepts the request before it reaches `httpbin` and rejects it based on the `AuthorizationPolicy` method rule.

Check SPIFFE identity in ztunnel logs on the remote cluster:

```bash
kubectl logs -n istio-system -l app=ztunnel \
  --tail=25 --context k3d-mesh-remote | \
  grep -E "spiffe|mtls|HBONE|peer"
```

Open Kiali (runs on the primary cluster):

```bash
kubectl port-forward svc/kiali \
  -n istio-system 20001:20001 \
  --context k3d-mesh-primary &

open http://localhost:20001
```

Navigate to Graph, select the `httpbin` namespace. The edge between curl and httpbin shows a mTLS lock icon. Click the edge and open the Security tab — Kiali shows the SPIFFE identities on both ends of the connection.

</Step>

## What changes, what stays the same

No annotations on pods. No `sidecar.istio.io/inject` label. No per-pod resource requests for Envoy. ztunnel costs approximately 70 MB per node, shared across every pod on that node.

What doesn't change: the `AuthorizationPolicy` spec is identical to sidecar mode. SPIFFE identity is the same trust model. Kiali observability works the same way. Existing sidecar-mode namespaces can run alongside ambient-mode namespaces in the same cluster — migration is per-namespace, not a cluster-wide flag day.

The setup script and all manifests are at `season1/ep08-istio-ambient-multicluster/` in the repo.
