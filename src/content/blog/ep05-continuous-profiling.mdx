Your P99 latency just spiked 40%.

Metrics show CPU usage is up. Traces confirm the request took longer — but all your spans look normal. Nothing in the logs explains it. You know *something* is slow, but you have no idea *where*.

This is the gap metrics, traces, and logs cannot fill. **Continuous profiling fills it.**

This post walks through deploying Parca and Parca Agent on a local k3d cluster. No SDK. No application changes. The eBPF-based agent profiles every container on the node automatically and ships CPU flame graphs to the Parca server.

---

## The Fourth Observability Signal

The three pillars give you:

| Signal | Answers |
|--------|---------|
| Metrics | *Is something wrong? How often?* |
| Traces | *Where in the request path?* |
| Logs | *What events occurred?* |
| **Profiling** | ***Which function is consuming the CPU?*** |

Profiling works by sampling the call stack of every running process every 10 milliseconds. Each sample captures the full chain of function calls at that moment. Aggregate thousands of samples and the hot paths become visible as a flame graph — the width of each block is proportional to CPU time.

**Why continuous?** Traditional profiling is triggered manually — you SSH into a box during an incident, run `perf` or `py-spy`, then turn it off. By then the spike may have passed, and you've lost data from before the incident. Continuous profiling always runs, at negligible overhead, so you can query historical profiles from before the problem started.

## Parca Architecture

```
k3d cluster: ep05-profiling
│
├── parca namespace
│   ├── parca (Deployment)       ← Server: stores profiles, serves flame graph UI on :7070
│   └── parca-agent (DaemonSet)  ← eBPF profiler: samples every process on the node
│
└── sample-app namespace
    └── sample-app               ← CPU-bound workload (generates visible flame graph)
```

**Parca Agent** uses eBPF to attach to the Linux perf subsystem at the kernel level. It:
1. Discovers processes by watching the cgroup hierarchy (maps profiles to k8s pod labels)
2. Samples stack traces at 10ms intervals via the kernel — zero user-space overhead
3. Uploads compressed profiles to the Parca server over gRPC on port 7070

**Parca Server** stores profiles in a time-series format and serves the query UI. You can select any label (pod name, namespace, container) and view the flame graph for any time window.

## Step-by-step

### 1. Create the k3d cluster

```bash
k3d cluster create ep05-profiling --agents 1 --wait \
  --k3s-arg "--disable=traefik@server:0"
```

`kubectl cluster-info --context k3d-ep05-profiling` should return without error.

### 2. Create namespaces

```bash
kubectl create namespace parca --context k3d-ep05-profiling
kubectl create namespace sample-app --context k3d-ep05-profiling
```

### 3. Install Parca Server

```bash
helm repo add parca https://charts.parca.dev
helm repo update

helm install parca parca/parca \
  --namespace parca \
  --kube-context k3d-ep05-profiling \
  --wait \
  --timeout=300s
```

```bash
kubectl rollout status deployment/parca -n parca --context k3d-ep05-profiling
```

`kubectl get pods -n parca --context k3d-ep05-profiling` should show `parca-*` in Running state.

### 4. Install Parca Agent

```bash
helm install parca-agent parca/parca-agent \
  --namespace parca \
  --kube-context k3d-ep05-profiling \
  --set config.remoteStoreAddress="parca.parca.svc.cluster.local:7070" \
  --set config.remoteStoreBearerToken="" \
  --set config.remoteStoreInsecure=true \
  --wait \
  --timeout=300s
```

The agent runs as a DaemonSet — one pod per node. On a single-node k3d cluster, one agent pod profiles every container on the machine.

```bash
kubectl get daemonset parca-agent -n parca --context k3d-ep05-profiling
```

```
NAME          DESIRED   CURRENT   READY   NODE SELECTOR
parca-agent   1         1         1       <none>
```

### 5. Deploy the CPU-bound sample app

```bash
kubectl apply --context k3d-ep05-profiling -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-app
  namespace: sample-app
  labels:
    app: sample-app
    parca.dev/scrape: "true"
spec:
  replicas: 1
  selector:
    matchLabels:
      app: sample-app
  template:
    metadata:
      labels:
        app: sample-app
        parca.dev/scrape: "true"
    spec:
      containers:
        - name: cpu-worker
          image: busybox:latest
          command:
            - /bin/sh
            - -c
            - |
              echo "Starting CPU load generator..."
              while true; do
                i=0
                while [ $i -lt 100000 ]; do
                  i=$((i + 1))
                done
                sleep 1
              done
          resources:
            requests:
              cpu: "100m"
              memory: "64Mi"
            limits:
              cpu: "200m"
              memory: "128Mi"
EOF

kubectl rollout status deployment/sample-app -n sample-app --context k3d-ep05-profiling
```

### 6. Start port-forward and open Parca UI

```bash
kubectl port-forward svc/parca 7070:7070 -n parca \
  --context k3d-ep05-profiling &
```

Open **http://localhost:7070** in your browser. Profile data takes 30–60 seconds to appear after the agent starts. If the label selector is empty, wait a moment and refresh.

## The Flame Graph (Hero Moment)

Open the Parca UI at http://localhost:7070:

1. **Select a label** from the dropdown — choose `{namespace="sample-app"}` or `{pod="sample-app-*"}`
2. **Choose a time range** — the last 5 minutes will show live data
3. **Click Query** — the flame graph renders

**How to read it:**

```
┌──────────────────────────────────────────────────────────────────┐  ← widest = hottest
│                   sh (busybox shell)                             │
├──────────────────┬───────────────────────────────────────────────┤
│  sleep           │             arithmetic loop body              │  ← second level
└──────────────────┴───────────────────────────────────────────────┘
```

- **Width** = CPU time (wider = more time spent here)
- **Stack grows upward** — the bottom is the entry point, top is the leaf function
- **Widest block at top** = the function to optimise first

For the busybox sample app, the tight `while` loop appears as the widest block — exactly the CPU work we scripted.

## Profile Diffing

Parca supports comparing two time windows — useful for validating performance fixes:

1. In the Parca UI, switch to the **Diff** tab
2. Select **Profile A** (before your change) and **Profile B** (after)
3. Red blocks = functions that got *slower*, Green blocks = functions that got *faster*

This makes it possible to prove — not just claim — that a performance PR reduced CPU usage.

## What You Built

```
k3d cluster: ep05-profiling
└── parca/
    ├── parca          ← Server + UI (flame graphs, profile diff, label queries)
    └── parca-agent    ← DaemonSet eBPF profiler (zero app changes)

sample-app/
└── sample-app         ← busybox CPU loop, profiled automatically
```

**Key facts:**

| Concept | Detail |
|---------|--------|
| eBPF overhead | < 1% CPU — safe to run always-on in production |
| Profile resolution | 10ms sample interval (100 samples/sec per process) |
| Data retention | Configurable; default in-memory for demo, use object storage for production |
| CNCF status | Sandbox project — active development, Grafana Pyroscope is the graduated alternative |
| No SDK required | Agent reads debug symbols from compiled binaries — works with Go, Rust, C, Python |

```bash
# Remove kubeconfig context

```

Or run the full teardown via the setup script:
